# StatusWidget

a circular (radial) progression bar. **right-click** opens a menu to choose what it shows **at rest** — **experience**, **reputation**, **honor**, **currency**, or **auto** (xp while leveling → reputation at max) — and what it swaps to while the cursor is over the bar **and Shift is held** (a configurable **secondary** value). defaults to auto at rest / honor on Shift, so it behaves exactly as before out of the box. the fill sweeps from the bottom clockwise inside a metal border. the hollow center is a **flourish slot** — it plays an Orbit animation on Great Vault unlock today, and is the home for future level-up / reward FX.

this is a self-contained plugin — it depends on no other plugin (plugins may not depend on other plugins). it reuses only Core: `OrbitEngine.Frame` (edit-mode persistence), `OrbitEngine.ColorCurve`, `OrbitEngine.SchemaBuilder` / `Config`, `OrbitEngine.Pixel`, `Orbit.EventBus`, and `PluginMixin` settings.

## blizzard xp/rep/honor bar

the orb replaces Blizzard's bottom XP/reputation/honor bar, so the plugin suppresses `StatusTrackingBarManager` whenever it owns the slot — in `OnLoad` (plugin enabled) and via a registered Blizzard hider (`Orbit:RegisterBlizzardHider`) for the PluginManager's **Both disabled** tri-state (plugin off, but the bar still hidden). it uses `OrbitEngine.NativeFrame:SecureHide` — the state-driver path the [EditMode README](../../Core/EditMode/README.md) documents for status tracking bars, combat-deferred via `CombatManager:QueueUpdate` since the driver can't be set in lockdown. the VisibilityEngine marks the bar `ownedBy = "Status Widget"`, so it drops out of the VE list while the plugin is enabled. because the plugin is `liveToggle`, a **profile switch** can disable the orb without a `/reload`, so `OnDisable` hands Blizzard's bar back — it drops the hide driver and re-shows the manager — **unless** the **Both disabled** tri-state (`Orbit:IsBlizzardHidden`) still wants it hidden. (the PluginManager tri-state toggle itself is `/reload`-gated; the live path is the profile switch.)

## great vault flourish

the Great Vault "objective complete" animation is an **insecure** `EventToast` (`Enum.EventToastDisplayType.WeeklyRewardUnlock` / `WeeklyRewardUpgrade`), queued through `C_EventToastManager`. with **Replace Great Vault Popup** on (Behaviour tab, default), `GreatVault.lua` wraps `EventToastManagerFrame.DisplayToast`: it replicates the original's advance-remove, drains vault toasts off the front via `RemoveCurrentToast` (firing `PlayVaultFlourish(info.title, info.subtitle)` for each), then displays the front via `orig(self, true)`. every non-vault toast (level-up, scenario, M+ keystone) is untouched. toggling the setting off restores Blizzard's popup live (the hook stays installed but no-ops).

the event text (`info.title` / `info.subtitle`) is Blizzard's own localized string for the event ("you unlocked vault slot…", etc.). for the **Upgrade** toast `info.subtitle` is an item link, which a FontString renders as the item name (Blizzard's own toast resolves it to an item level instead — a future refinement if wanted).

no secret/taint/combat concerns — the EventToast system is pure insecure UI. `C_WeeklyRewards.GetActivities()` (non-secret) is available for a future vault-progress display mode but is not used yet.

**click to open** — while a flourish is showing (`_flourishActive`, set in `PlayVaultFlourish`, cleared when the held frame fades), a left-click on the orb calls `OpenGreatVault` → `WeeklyRewards_ShowUI()` (Blizzard's own remote-open path, used by the map POI / Encounter Journal). outside the flourish window the click does nothing. the FX art is recentred in its cell via `VAULT_FX_OFFSET_X/Y`.

## social toasts

`SocialToast.lua` gives Battle.net friend, session-time, and voice toasts the same treatment as the vault. they all inherit `SocialToastTemplate` and display through the global `AlertFrame_ShowNewAlert(frame)` *after* their FontStrings are populated. with **Replace Social Toasts** on (Behaviour tab, default), a **`hooksecurefunc`** post-hook recognises those frames by structure (`TopLine`/`BottomLine`/`CloseButton`), captures their already-localised text (preserving Blizzard's embedded colors) + icon, then `DismissAlert`s the just-shown toast (stop its anims + hide, exactly like a right-click) before it renders, and calls `PlaySocialFlourish`. all other alerts pass through untouched. the BN sound still plays (it fires in `ShowToast` before the alert).

**coverage** — every BN toast type funnels through the one `BNToastFrame` (inherits `SocialToastTemplate`), so the structural match catches them all: friend **online/offline**, **broadcast** (custom status), **pending-invites** (login summary), **new friend request**, **club invitation**, and **club-finder approved** — plus the session-time and voice-prompt frames (same template). we replay whatever lines/icon Blizzard rendered, so each type carries its own correct text. (online/offline early-out for `isCompanionApp`, so mobile-app logins never toast — nothing to replay.)

**CVar** — the replacement only works if Blizzard still *generates* the toasts, so `EnforceToastCVar` (called from `ApplySettings`, i.e. on login and on any settings change) forces the master CVar `showToastWindow` on whenever **Replace Social Toasts** is set. that CVar gates the whole BN toast system but is read only once into `SetToastsEnabled` at `VARIABLES_LOADED` with no live callback — so we also call `BNToastFrame:SetToastsEnabled(true)` to re-register the events live (otherwise it'd need a `/reload`). per-type prefs (`showToastOnline`, …) are left to the user.

**taint:** this MUST be a `hooksecurefunc`, never a global replacement. assigning `AlertFrame_ShowNewAlert = ourFn` taints the Blizzard global; since secure code (EditMode's party-frame refresh → `CompactUnitFrame_UpdateHealthColor`'s secret colour compare) reads that environment, it throws "execution tainted by Orbit". the post-hook is isolated, so suppression stays taint-free.

online toasts set the character-name line async after the alert call, so the replay shows account name + state without the character line — acceptable.

`PlaySocialFlourish` = glow (battle.net blue) + the toast's **own icon in the centre** + side text, no vault FX and no click-to-open-vault. `CaptureIcon` lifts the toast's `IconTexture` (BN logo / game / invite — a sprite-sheet region) by replicating its `GetTexture()` + `GetTexCoord()` onto `frame.SocialIcon`; the dark backdrop shows behind it. lingers `SOCIAL_HOLD` (8.5s).

## mail toast

`Mail.lua` toasts new mail. `UPDATE_PENDING_MAIL` fires whenever pending-mail state changes, so it only toasts on a **false→true** `HasNewMail()` transition (tracked via `_hadMail`) — re-fires while mail sits unread stay quiet. `GetLatestThreeSenders()` gives the names; the toast is `PlayMailFlourish("New Mail", "From <names>")`. `PlayMailFlourish` = warm glow + the **minimap mail reminder flipbook** (`UI-HUD-Minimap-Mail-Reminder-Flipbook-2x`, 3×4 / 12 frames, `MAIL_FX_SIZE`, looped for `MAIL_HOLD` (6.5s) via a cancelable `C_Timer`, no held frame) + side text. toggle `ShowMailToast`; test `/orbitmail`.

## generic flourishes + the new events

The centre FX hub gained two generic renderers the new modules reuse (so each event is just a colour + atlas/icon + text fed to the queue): `PlayBurst(atlas, color, title, sub)` (a tinted celebration atlas on `frame.BurstFX` that pops + fades — level-up `AftLevelup-WhiteStarBurst`, renown `evergreen-toast-celebration-star`, boss-defeat `BossBanner-SkullCircle`) and `PlayIconFlourish(icon, color, title, sub, coords)` (a disc-masked icon reusing the loot disc — collectibles / recipe / spell / etc.). Per-kind glow colours live in `Plugin.FlourishColors` (gold / collect / arcane / defeat / honor / rep) so the modules don't duplicate the constants. New event sources: **Milestones** (level-up / renown, event-listen), **GreatVault** also drains **spell-learned** (`eventType` 21) and plays the **vault-upgrade** flipbook (`greatVault-anim-upgrade-FX`, re-pointed via `frame.FlourishFlip`), **Loot** enqueues a **boss-defeat** burst on `BOSS_KILL` (before the reel), and **AlertToasts** replays the reward alerts. Each toast category is gated (`ShowMilestones`, `ShowRewardToasts`).

## fill modes + idle centre

the ring's data source is the **primary / secondary** selection (see *source menu* and *mode resolution*) — `BuildRecord` dispatches the resolved mode to its record builder, including `CurrencyRecord` (non-secret). `CurrencyRecord` shows the currency the source menu picked for the active slot (`ResolveMode` stashes `_activeCurrencyID`), else the first backpack-tracked currency; a currency with **no known cap** isn't trackable, so it falls back to the auto (xp/rep) source rather than fabricating a max. it flourishes on a gain while currency is the primary source.

**Durability is NOT a fill source — it's an always-on warning.** Whenever equipped-gear durability (`GetInventoryItemDurability` summed, `_DurabilityPct`) drops into the warn band *and the orb is idle*, the whole **hollow centre fills with cracked metal** — durability is gear wearing out, so the fracture is the metal frame breaking down (not glass). Tinted **yellow ≤40% (slow pulse) → red ≤20% (faster pulse)** — WoW's own damaged→broken convention — with the **durability %** (white + a hard dark shadow so it reads over the bright metal) in the centre. The art is a **6-variant cracked-metal set** (`orbit-radial-crackedmetal-{1..6}.tga`): a **Voronoi fracture** filling the whole disc (seeds spread area-uniform across the centre, one owning the middle), each cell an **opaque forged plate** lit as a tilted facet under a single light (some plates bright, some in shadow → genuine **3D depth**) and slightly domed, with a soft forged sheen on the light-facing edge, ambient-occlusion shadow on the recessed edge, a hammered surface, and deep dark **cracks** between plates. **Opaque is the key** — the shading is baked into the texture's **RGB** (greyscale, floored above zero), and **alpha carries only the plate shape**, so plates render fully solid and the cracks are crisp transparent gaps. (Baking the shading into *alpha* instead made shadowed plates see-through, which read as glass — the bug that took several passes to spot.) **Layering:** the disc lives on `frame.Content` at **BACKGROUND sublevel −2 — beneath the ring/border/background** (the track is BACKGROUND 0), so the annular ring art frames it and covers its outer edge while it shows through the transparent-centred ring in the hollow. (Earlier glass passes — uniform pie-wedges, a forensic *spiderweb*, then faceted glass shards — were dropped: glass is the wrong metaphor for armour wear; cracked metal coheres with the orb's metal border and reads as equipment breaking down.) `_RandomizeCrackedMetal` picks a **random variant + a random rotation** (`Texture:SetRotation`) on each fresh warning (and at `SetupFillModes`, so it varies per `/reload`) — re-picked only on the `_durabilityWarn` false→true transition, so it never flickers mid-warning. It does **not** use the toasts' inner-border vignette (`_RefreshInner` is flourish-only now) — the opaque cracked disc *is* the centre backdrop. The warning pulse floor is kept high (`0.62→1.0`) so the now-opaque metal throbs without fading see-through. Refreshed on `UPDATE_INVENTORY_DURABILITY` and every `UpdateBar`, regardless of what the ring shows. The two idle-centre layers (durability % and the optional `ShowCenterNumber` numeral) yield to any flourish (`_ClearCenterFX`) and arbitrate (durability % > numeral, via `_durabilityWarn`). A left-click with no flourish opens the panel the orb shows (honor → PvP UI, currency → the character sheet's currency tab, else the paperdoll) via `OpenForMode`.

**durability break flourish (40% / 20%)** — `_UpdateCrackedMetal` tracks `self._lastDuraPct` and, on each **downward crossing** of `DURA_WARN` (40%) or `DURA_CRIT` (20%), fires a one-shot `PlayShatterFlourish(broken)` through the queue (so it never interrupts an active flourish, and the first read on login only seeds the tracker — no false fire). `_RenderShatter` is as weighty as the level-up: **metal shrapnel bursts outward** (`orbit-radial-shardburst.tga`, additive, scale 0.30→1.85 + fade, tinted amber at 40% / red at 20%) plus the full shared impact set (`BorderFlash` + `_RingImpact` + `WidgetShake`) — no side text. The **inner cracked circle forms mid-blast**: a `C_Timer.After(SHATTER_FORM_DELAY=0.30s)` (≈ the middle of the 0.7s burst, guarded on `self._event == "shatter"`) reveals the persistent cracked-metal warning via the shared `_ShowCrackedWarning(crit, pct)` helper, as if the explosion leaves the broken surface behind. Built in `_BuildShatterFX`; the shrapnel is distinct from the soft level-up rays. **`/orbitshatter`** previews both breaks (40% amber, then the 20% red queued behind it); `/orbitshatter 40` or `20` fires just one. It works on full-durability gear — `_ShowCrackedWarning` substitutes a representative band % when the real durability isn't low, so the test reads sensibly.

**durability tooltip** — `ShowTooltip` → `_AppendDurabilityTooltip` adds a per-slot breakdown **only when gear is actually damaged**, mirroring the Durability datatext: a header (reusing `PLU_DT_DURABILITY_TITLE`), each damaged equip slot (`GetInventoryItemDurability` over the 10 durable slots → localized `_G["HEADSLOT"]` names) with its % coloured by severity, then the **repair cost** (`GetRepairAllCost` → `GetCoinTextureString`, reusing `PLU_DT_DURABILITY_REPAIR_COST`). No cross-plugin dependency — it doesn't call the Datatexts module, just the same WoW APIs.

**cracked ring (durability-gated)** — a crack overlay (`frame.RingCrack`) on `borderHost` (ARTWORK sublevel 1, **above** the fill + border), normal BLEND with dark fracture lines that darken along the cracks. Shown **only when durability is low** and scaled by severity via `_SetRingCrack(level)`: off above 40%, **light** (`orbit-radial-ringcrack-light.tga`, a few cracks) at 20-40%, **heavy** (`orbit-radial-ringcrack-heavy.tga`, more cracks + chips — a *superset* of light) below 20%. Driven from `_UpdateCrackedMetal` by the **actual** durability (so the cracks persist through centre flourishes, unlike the cracked centre which yields), idempotent so frequent `UpdateBar`s don't re-texture. Each time the cracks **first appear** (0 → shown) a fresh **random `SetRotation`** spins them so they never look the same per load / episode; a light↔heavy change keeps the same spin so the heavy superset's shared cracks stay put. The shatter reveal also sets it so `/orbitshatter` previews it on full-durability gear.

**marquee flourishes have weight**: every big flourish (vault unlock/upgrade, renown, boss-defeat, and the level-up) fires a shared impact set — a coloured **gold/mode ring flash** (`BorderFlash`, an additive tinted ring over the rim), **`_RingImpact`** (a hand-tweened scale punch-bounce on `frame.Content` that overshoots then settles + a decaying rotational shake on the ring art) and a **whole-orb shake** (`WidgetShake`, a Translation oscillation on `Content` that auto-reverts and composes with the reveal). So the orb *reacts* — flashes, punches, and shakes — instead of flashing a flat sprite.

**old → new number stomp** — as the production opens, the **old** level / renown number (`frame.OldNumber`, dim grey) **fades in quick** at centre (`OldNumberInAnim`, 0.18s). Then the **new** number **stomps it out** the instant it lands: `_LevelImpact` (fired on the slam for level-up, or after a brief lead — `MILESTONE_GROW_LEAD` — for the renown grow-in, so the old renown number gets its moment first) plays `OldNumberOutAnim` — a quick **crush-down + drop + fade** (Scale → 0.35, Translate down, Alpha → 0 over 0.22s) — while the new number's own tween drives the new value over the top. The old value is `level − 1` for level-up and the event's `oldRenownLevel` for renown (so it's always the real previous number). `OldNumber` is created before `LevelNumber` so the new number draws on top; `_ClearCenterFX` resets it between flourishes.

**milestones are a production** — **level-up**, **major-faction renown** and **honor (PvP)** level-up share one centre production (`_RenderMilestone`, in `StatusWidget.lua`; driven by `Milestones.lua`): a **light burst** + radiating **rays** (additive, counter-spinning), an optional rising **up-arrow** (`npe_arrowupglow`), and the big **centre number** (`textSize` 40), plus the full impact set above — all **tinted per milestone**. **Each milestone brings its own sprites** (`MILESTONE_FX_LEVELUP` / `MILESTONE_FX_RENOWN` / `MILESTONE_FX_HONOR`, applied in `_RenderMilestone`; the up-arrow is gated on `_lvlShowArrow` in `_LevelImpact`):
- **level-up** — the authentic level-up sheet: `aftlevelup-whitestarburst` burst + `aftlevelup-lines1` rays + the up-arrow (a leveling cue). Gold, the level number, no side text (`_RenderLevelUp`).
- **renown** — a sparkly **starburst** (`ArtifactsFX-StarBurst`, the Artifact power-up FX — a radiating "starfield" look, distinct from the level-up whitestarburst) + the evergreen renown-toast **shine** rays (`evergreen-toast-celebration-shine`), no up-arrow. Rep colour, the renown level, the **faction name beside the orb** (`_ShowFlourishText`; localized from `C_MajorFactions.GetMajorFactionData`) (`_RenderRenown`).
- **honor** — the honor-system **prestige flash** burst (`honorsystem-bar-rewardborder-prestige-flash`, a radiant honor-prestige sprite) + the PvP **rank glow** rays (`pvpqueue-rankglow`), no up-arrow. Honor **red**, the honor level, **no side label**, the **slam** motion for a martial beat (`_RenderHonor`). Driven by `HONOR_LEVEL_UPDATE` — only a real level increase flourishes (honor-xp gains don't), and the first update just seeds the baseline so there's no flourish on login.

So the three read as distinct celebrations rather than recolours of one. Most of the FX is built in `_BuildLevelUpFX`; the number is choreographed by a **manual tween** (`_PlayMilestoneNumber`, same OnUpdate-driver pattern as `_RingImpact`, not an AnimationGroup — the sequence needs position + scale + alpha threaded together) with **two motions**:

- **slam** (level-up): the number **winds up** (floats up + fades in), then **slams down** to its lowest point and **grows** on impact (`LVLNUM_GROW_PEAK`) before **dissolving while drifting larger** (`LVLNUM_GROW_DRIFT`). **Timing is the point:** the flash detonates *on the slam* — the tween fires **`_LevelImpact`** (glow + burst + rays + up-arrow + ring flash + `_RingImpact` + orb shake) on the exact frame the number reaches its lowest point (`t >= LVLNUM_LAND`), so the number is most-downward *precisely as the full-screen flash goes off* — frame-perfect.
- **grow** (renown): no rise, no slam, no bounce — after a brief **old-number lead** (`MILESTONE_GROW_LEAD`, so the old renown number shows first) the burst fires and the number **scales up from small** (`LVLGROW_FROM`), **holds** at peak, then **dissolves while drifting larger** — reusing `LVLNUM_GROW_PEAK`/`DRIFT` so its size matches the level-up. Phase end-times are the `LVLGROW_*` constants.

`_RenderMilestone` *preps* the FX (atlas/colour, tints the number) and starts the tween; the `LVLNUM_*` / `LVLGROW_*` constants are the phase end-times. It scales around dead-centre so the grow reads from the middle of the orb.

## flourish queue (timing)

`FlourishQueue.lua` is the single serialization point — **every** centre flourish (vault/social/mail/loot/…) goes through `Plugin:Enqueue(req)` instead of playing immediately, so events never overwrite each other. A request is `{ kind, render = function(plugin), selfPaced = bool? }`; the public `Play*Flourish` entry points enqueue, and the queue calls the matching `_Render*` after `_EnterEvent` has cleared the centre (`_ClearCenterFX`).

Timing contract: each flourish holds the centre for at least **`BUFFER` (3s)**; if nothing is waiting it extends to **`IDLE_HOLD` (5s)** — the extra time is the "end animation" (the linger + fade). The **end animation never plays while a request is queued**: at the 3s mark, if the queue is non-empty the next flourish hard-cuts in (`_FqAdvance`), and an event that arrives *during* the idle linger cuts it. The loot reel is **`selfPaced`** — it runs its own item sequence and calls `_FqBurstDone()` when the items drain, handing the buffer/linger back to the queue (so loot is never interrupted by, nor interrupts, another flourish). `_EnterEvent`/`_ExitEvent` are queue-driven (enter on pop, exit when the queue empties).

**unified end-fade** — at the fadeout phase the side text (`FlourishTextOut`) **and the whole hollow centre** fade out together in lockstep (both `0.5s`). The centre is faded as one group via `frame.CenterFadeOut` (an Alpha group on `frame.Center`, `SetToFinalAlpha`), so the inner-border vignette, the toast icon and any FX all dissolve at the same time — previously the text faded first, the icon hard-popped, and the vignette faded *after* (the social-toast inconsistency). `_EnterEvent` resets `frame.Center` alpha to 1 (clearing any prior fade); `_ExitEvent` stops the group, **snaps the vignette hidden** (so the alpha reset doesn't pop it back, and `_RefreshInner` via `UpdateBar` is a no-op), `_ClearCenterFX`es the children, then resets the alpha. The text frame is parented to the orb (not `frame.Center`), so it fades only via `FlourishTextOut` — no double-fade.

## center FX hub

`SetupCenterFX` builds three things, played together by `PlayVaultFlourish(title, subtitle)`:

- `frame.Flourish` — an additive glow underlay (`orbit-radial-glow.tga`, tinted gold), quick scale + alpha pulse (`frame.FlourishAnim`).
- `frame.FlourishFX` — Blizzard's **authentic** unlock FX: a texture on the `greatVault-anim-unlocked-FX` atlas driven by a `FlipBook` animation (9 rows × 9 cols, 77 frames, 2.57s — the exact params from `EventToastManager.xml`). `frame.FlourishFXAnim`.
- `frame.FlourishFXFinal` — a **separate static** texture that holds the final frame during the linger. needed because a FlipBook reverts the flipbook texture's texcoords *after* `OnFinished`, so the live texture can't be pinned (the whole sheet flashes). `HoldVaultFX` copies the flipbook's exact texcoords — `FlourishFXFinal:SetTexCoord(FlourishFX:GetTexCoord())`, read in `OnFinished` before the revert — onto this static texture, so the held frame is byte-identical to where the burst ended (no cell math, no jump). `frame.FlourishFXHold` then lingers `FLOURISH_HOLD` (5s) and fades.
- `frame.FlourishText` — a `Title`/`SubTitle` FontString block just right of the ring (`FLOURISH_TEXT_GAP`), in the **global Orbit font** via `ApplyFlourishFont` (`Orbit.Skin:SkinText`, re-applied on theme change in `ApplySettings`). single line, `SetWordWrap(false)`, no fixed width → auto-sized, never wraps. vertically centered (`PlayVaultFlourish` re-anchors: one line on the orb's middle, two lines straddle it). holds for `VAULT_FX_DURATION + FLOURISH_HOLD` to linger with the held frame.

sizing knobs: `FLOURISH_SIZE` (glow), `VAULT_FX_W/H` (atlas), `FLOURISH_TITLE_SIZE` / `FLOURISH_SUB_SIZE` (text), `FLOURISH_TEXT_GAP` (distance from ring), `FLOURISH_HOLD` (linger seconds). the text parents to the orb frame, so it scales with `Scale`.

it's a hub — add new center animations / flourish types here as they arrive. the text block parents to the orb frame, so it scales with `Scale` (tune `FLOURISH_TEXT_*` if you want it unscaled / repositioned).

**`/orbitvault`** — dev/test command (in `GreatVault.lua`) that fires `PlayVaultFlourish` with sample text (`PLU_SB_V2_VAULT_TEST`) so you can preview the FX + side text. no-ops if the plugin is disabled (no frame); the orb must be visible to see it.

## source menu

**right-click** the orb to open a `MenuUtil.CreateContextMenu` (`SourceMenu.lua`): the plugin name as a title, then two nested radio submenus — **Show at Rest** (`PrimarySource`: Auto / Experience / Reputation / Honor / Currency) and **Shift Shows** (`SecondarySource`: None / Experience / Reputation / Honor / Currency). selecting a radio writes the setting and `UpdateBar`s live. the same two settings appear as Behaviour-tab dropdowns, so the menu and the config panel stay in sync (both write `PrimarySource` / `SecondarySource`). the menu is suppressed in Edit Mode.

the **Currency** entry breaks out into its own picker submenu — `_EligibleCurrencies` (in `FillModes.lua`) lists the player's discovered currencies that have a **known cap** (`maxQuantity`, else `maxWeeklyQuantity`), since a currency with no max can't fill a radial. picking one sets that slot's source to `currency` **and** records the currency id in a per-slot key (`PrimaryCurrencyID` / `SecondaryCurrencyID`), so the at-rest and Shift slots can each track a different currency. when no currency qualifies the submenu shows a disabled "no currencies with a cap" line. (the list mirrors the character-pane currency tab, so a currency under a collapsed header is absent — expand it there to pick it.) the Behaviour-tab `Currency` dropdown option just enables currency mode using the first backpack-tracked currency; the specific currency is chosen here.

## mode resolution

the orb shows a **primary** source at rest and swaps to a **secondary** source while the cursor is over it **and Shift is held** — both chosen from the source menu (defaults: primary `auto`, secondary `honor`, preserving the original behaviour). `ResolveMode()` resolves the active one each update:

1. **secondary** (hover + Shift) — if `_hovered` AND `IsShiftKeyDown()` and `SecondarySource ~= "none"`, resolve it. driven by `OnEnter`/`OnLeave` plus a `MODIFIER_STATE_CHANGED` subscription that re-resolves while hovered, so pressing/releasing Shift over the bar flips it live.
2. **primary** otherwise — resolve `PrimarySource`.

`_ResolveSource(source)` maps a source key to a concrete mode: `xp` / `rep` / `honor` / `currency` map directly, and `auto` maps to **XP** while `UnitLevel < GetMaxPlayerLevel()` (xp not user-disabled) else **Reputation**. `BuildRecord` then dispatches the resolved mode to its record builder (`XPRecord` / `BuildRepRecord` / `HonorRecord` / `CurrencyRecord`); each returns the same `{mode, name, level, current, max, color[, rested]}` shape.

- **Reputation** handles major-faction renown, paragon, and standard reaction thresholds; spans are plain non-secret numbers reduced to a 0-based `current/max`. the whole rep bar (renown / paragon / standing) uses one configurable `RepColor`.
- **Currency** shows the currency picked in the source menu for the active slot (`_activeCurrencyID`), else the first backpack-tracked one (`CurrencyRecord`, in `FillModes.lua`). only currencies with a known cap are trackable; without one it falls back to `_AutoRecord` (xp/rep).

while in **XP** mode the orb also draws a **rested-XP band**: `frame.RestedFill` is a second swipe sitting one frame level *below* the main fill, set to `(current + GetXPExhaustion()) / max` in `RestedColor`; the main fill paints over its current portion so only the rested band (current..current+rested) shows. `GetXPExhaustion` is `issecretvalue`-dropped (xp goes secret in encounters), and the layer hides outside XP mode.

> **prey hunt — removed (to revisit).** An earlier version drove the orb from the Midnight prey hunt (`PreyHuntProgress` UIWidget + quest objective, red fill, centre target icon, Blizzard-widget suppression, chat "hunted/kill" prompts). It was pulled entirely because reading the live widget payload / scanning monster chat during an active hunt tainted Orbit's shared execution and surfaced as `CompactUnitFrame_UpdateHealthColor` "execution tainted by Orbit" on EditMode-enter in a prey zone. Revisit only with a taint-safe design (Preydator's approach: read stage from quest APIs, treat the widget Setup as a freshness signal only, never read live widget fields in runtime paths).

## loot reel

`Loot.lua` replays boss-kill and personal loot as a **sequential reel** through the centre: each item's icon pops in (`frame.LootIcon`, **circular-masked to fill the hollow centre** — a `CreateMaskTexture` disc using Blizzard's portrait alpha mask, sized `LOOT_ICON_RATIO` = `CENTER_RATIO`, bound to the texture so it survives the per-item `SetTexture` swap) with a quality-coloured glow (`_PlayGlow(item.color)`), its name beside the orb in its quality colour, a stack count (`frame.LootCount`) tucked inside the disc when >1, then the next item slides in. `PlayLootReel(items)` appends + starts; `_LootReelStep` is a `REEL_STEP` timer chain — while items remain each step interrupts the previous item's side-text before it auto-exits, and when the queue drains the last item's text finishes naturally and fires `_ExitEvent`. a non-loot event mid-reel calls `_CancelLootReel` (drops the queue); the `Animation` reveal/conceal wraps it like any other event.

**two sources**, both gated by **Replace Loot Banner** (default on):

- **encounter loot (yours only)** — a dedicated frame (never the shared EventBus) listens for `BOSS_KILL` + `ENCOUNTER_LOOT_RECEIVED` (party/raid only) and reels **only the local player's** drop, not the whole party's: `IsLocalPlayerLoot(playerName)` gates it (matches the short name, plus the realm when the event name is `Name-Realm`). Your own loot shows **no winner line** (it's you). args follow Blizzard's BossBanner — `encounterID, itemID, itemLink, quantity, playerName, className` — **not** the auto-generated docs, which mis-label slots 5-6 as `itemName/fileName`. (The class-coloured winner-name path is kept for the `/orbitloot` preview and a possible future "show party loot" toggle, but is unused live.)
- **personal loot** — `SHOW_LOOT_TOAST` (items) + `SHOW_LOOT_TOAST_UPGRADE`, reeled only **outside** a 6s `ENCOUNTER_WINDOW` after encounter loot, so your own raid drop isn't shown twice (the group reel already has it). this is what covers world / solo / bonus-roll loot.

**suppression** is taint-safe (mirrors the social fix — never a global replace):

- `hooksecurefunc("BossBanner_Play", …)` → `BossBanner_Stop` + `TopBannerManager_BannerFinished`. fully replaces Blizzard's top banner (kill + loot); its "boss defeated" sound still plays. capture is independent of suppression (our own event frame), so the reel works even if suppression is off.
- `hooksecurefunc("AlertFrame_ShowNewAlert", …)` → `DismissAlert` on personal item-loot frames (`LootWonAlertFrame` / `LootUpgradeFrame`); currency (`frame.isCurrency`) and money toasts pass through untouched. invariant: we only hide a toast we either reel ourselves or whose item the group reel already covers.

item data resolves via `Item:CreateFromItemLink(...):ContinueOnItemLoad` (icon / name / quality), so uncached items still render. loot fields are **non-secret** — no `issecretvalue` guards. **`/orbitloot`** plays a sample defeat banner, then reels a quality spread + a stack count (no winner lines — your-own-loot, matching live).

## loot rolls

`LootRoll.lua` replaces Blizzard's group-loot roll UI with styled panels attached to the orb. A dedicated frame captures `START_LOOT_ROLL` (→ add a panel), `CANCEL_LOOT_ROLL` (→ remove one), and `CANCEL_ALL_LOOT_ROLLS`. Each panel shows the item as a **circular disc** (same mask as the reel) + stack count, the quality-coloured name, a **timer bar** drained each frame from `GetLootRollTimeLeft`, and **Need / Greed-or-Transmog / Pass** buttons using Blizzard's own glyph atlases (`lootroll-toast-icon-*`) inside an Orbit-themed panel. **Background, border and fonts all inherit the global theme** — `_StylePanel` (run on creation and re-run on every fill, so a live theme change shows on the next roll) applies `Skin:GetBackgroundColor` (bg, registered as a masked surface so it rounds), `Skin:SkinBorder` (the global border style + colour + size), and `Skin:SkinText` (global font on the name + rolled number); the timer uses `Skin:SkinStatusBar` (global bar texture). Per-roll colours — quality-coloured name, green/red roll number — are set after and preserved (SkinText only touches font).

- **roll types** come from the button id, matching Blizzard: **Pass 0, Need 1, Greed 2, Transmog 4** (disenchant 3 has no button). Greed and Transmog share one slot — Transmog shows iff `canTransmog`. `RollOnLoot` is unprotected, so the buttons work in combat; Need on a BoP fires Blizzard's own `CONFIRM_LOOT_ROLL` StaticPopup automatically. Clicking a button marks the panel rolled (buttons disabled) and waits for `CANCEL_LOOT_ROLL` to clear it.
- **dice reveal** — on `MAIN_SPEC_NEED_ROLL(rollID, roll, isWinning)`, `_PlayRollAnim` plays Blizzard's authentic FlipBook dice (`lootroll-animdice` 9×5/44) → glow burst (`lootroll-animdiceglow` 10×5/50, scale-up + fade) → the rolled **number** (green winning / red losing), with the positive/negative roll sound. The panel chrome (background, border, timer) drops for the reveal so the dice + number read against the world, leaving just the item disc + name; it's restored when the pooled panel is reused. One animation group per texture (auto-targeting, like the vault FX); the number group's `OnFinished` clears the panel, and a `CANCEL_LOOT_ROLL` arriving mid-reveal defers to it (`_animating` guard). The `/orbitroll` preview triggers the same reveal with a random roll when you click **Need**.
- **stacking + anchor** — `_LayoutRolls` stacks one panel per active roll off the orb, **screen-edge aware**: the first attaches to the orb corner toward screen-centre (orb top-left of screen → panels grow right + down), the rest stack in the grow direction.
- **suppression** (gated by **Replace Loot Roll**, default on) — **`HookScript("OnShow")` on each `GroupLootFrame1-4`** (alpha-0 + mouse-off, restored to alpha-1 when the setting is off). It must be `HookScript`, **not** `hooksecurefunc("GroupLootFrame_OnShow")`: the XML bound each frame's `OnShow` to the *original function reference* at load, so replacing the global is bypassed and never fires (this differs from `AlertFrame_ShowNewAlert`, which is invoked by name from Lua and so *can* be `hooksecurefunc`'d). `GroupLootContainer_Update` *is* called by name, so `hooksecurefunc` there collapses the bottom-managed container's layout slot. Capture is independent of suppression.
- **Bonus Roll** ("spend a coin to roll for extra loot") is **taken over** as one of our roll panels and Blizzard's suppressed. It rides `SPELL_CONFIRMATION_PROMPT` (`confirmType == Enum.ConfirmationPromptUIType.BonusRoll`, carrying `spellID / duration / currencyID / currencyCost / displayItemID`) and closes on `SPELL_CONFIRMATION_TIMEOUT`. `_ShowBonusRoll` morphs a pooled panel into bonus mode: the **bonus chest** (`BonusLoot-Chest` atlas) or the specific item disc, the **"Bonus Roll"** name (`BONUS_LOOT_LABEL`), a **dice Roll button** + Pass, and the duration timer (`_bonusEnd`, visual; the TIMEOUT event closes it). A bonus panel **always borders gold** (`BONUS_BORDER_COLOR`) regardless of theme: `_StylePanel` keys off `panel._isBonus` and forces the flat pixel border path (`forcePixel`, `>=2px`) because the rounded/edge-file styles ignore an explicit border colour — so the gold reads under any border style, even a borderless theme. Roll / Pass call the **same insecure `AcceptSpellConfirmationPrompt` / `DeclineSpellConfirmationPrompt`** Blizzard's own buttons call (combat-safe, like `RollOnLoot`) — no SecureActionButton needed. Blizzard's `BonusRollFrame` is a UIParent child merely *anchored* to the container (so the container-hide doesn't hide it); it's suppressed the same way as the group-loot frames — `HookScript("OnShow")` → alpha-0 + mouse-off (frame + prompt Roll/Pass buttons). All strings are WoW globals, so no new Orbit keys. `/orbitbonus` previews it.
- roll fields are **non-secret**. **`/orbitroll [itemID]`** previews a non-live panel (faked countdown, buttons just dismiss) so the styling is visible without a group.

## center state machine

the centre is a small priority state machine, separate from the radial fill:

- **transient events** (`_event` = `vault`/`social`/`mail`/`loot`) own the centre. `_EnterEvent` clears any centre icon, stops any running mail FX, drops any running loot reel (`_CancelLootReel`), and **fades the inner-border vignette in**; the flourish plays; when the **text fade finishes** (the last thing to finish in any flourish) `_ExitEvent` clears `_event`, hides the centre icon, **fades the vignette out**, and `UpdateBar` refreshes the fill. `_FadeInner(show)` cross-fades from the current alpha so a fast back-to-back event doesn't snap. a `loot` event is special: it holds `_event` across the whole reel (multiple items), exiting only when the queue drains.
- the **radial fill** is independent — `ResolveMode` (xp/rep/honor) keeps driving it during an event (e.g. the xp fill stays under a vault burst).
- the **center icon** is the social-toast icon (`PlaySocialFlourish`) or the loot reel's item icon (`frame.LootIcon`, cycled by `_LootReelStep`).
- `_ClearCenterFX` is the single "reset to a clean slate" call (run on every `_EnterEvent` and `_ExitEvent`): it stops/hides **every** centre layer + anim + the level-up driver and resets the transient flags (incl. the **side text** — parented to the orb, not `Center`, so the `CenterFadeOut` group never touches it — so a following flourish with no side text doesn't keep the prior one's).

## live-toggle teardown

the framework's live disable (`Core/Init.lua` `LiveTogglePlugin`) only hides the orb frame and cancels `plugin.timer`; it can't reach this plugin's raw `C_Timer`s, its three **UIParent-parented** OnUpdate drivers (`_impactDriver` / `_lvlNumDriver` / `_animDriver`), its four aux event frames, or the permanent `hooksecurefunc` suppression hooks. so `Plugin:OnDisable` quiesces the centre itself — cancels the queue + loot timers, `_ClearCenterFX`, hides **and nils** the three drivers (so they rebind to the rebuilt frame on re-enable), and resets the queue/event state — and sets `self._disabled`. while set, `Enqueue` no-ops (no centre flourish fires) and the BossBanner / AlertFrame / GroupLoot suppression hooks pass Blizzard's UI through (Loot/LootRoll via their `IsEnabled()`, Social/AlertToasts inline). `OnLoad` re-runs on re-enable and clears `_disabled`.

## center backdrop

**inner-border vignette** (`frame.InnerBackdrop`, `orbit-radial-innervignette`) — the sole centre backdrop, behind the icon/FX. a radial gradient that **peaks at the inner-border radius (~0.68) and fades to clear at the centre** (and out by ~0.80 so it never paints the fill band), tinted to the metal border's inner-edge hue (`INNER_BACKDROP_COLOR`, the sampled ~RGB 40/37/44 brightened). it **fades in for the life of any event** (vault/social/mail) and out at the end, at `INNER_BACKDROP_ALPHA` peak — a colour wash that ties the centre to the ring during a flourish. lives on `frame.Center` but sized to full `BASE_SIZE` so its peak lands on the real inner border (`frame.Center` doesn't clip). it's translucent and clear at the centre, so the world shows through behind icons/FX — the old near-opaque dark mask (`orbit-radial-centerbg`) was removed in favour of this single fading layer.

## screen-aware text

`_ShowFlourishText` anchors the side text away from the screen edge: orb on the **left** half → text grows **right**; on the **right** half → grows **left** (right-aligned). computed per-flourish from `frame:GetCenter()` vs the screen midpoint (scale-corrected), so it follows the orb wherever it's placed.

## how the radial fill works

the fill is a paused `Cooldown` swipe (the same mechanism Blizzard uses for the renown radial in `Blizzard_Journeys`), not a pre-rendered animation:

- `frame.Track` — static dark groove (BACKGROUND), the unfilled state
- `frame.Fill` — a `CooldownFrameTemplate` with `SetSwipeTexture(orbit-radial-fill)`, revealed by `CooldownFrame_SetDisplayAsPercentage(fill, current/max)` and tinted per-mode with `SetSwipeColor`
- `frame.Border` — static metal border drawn **above** the fill so the sweep can't spill over the rim
- `frame.Center` — empty child frame sized to `CENTER_RATIO` of the diameter; reserved for level-up content

`SetRotation(math.pi)` + `SetReverse(true)` align the engine's reveal to the asset's baked bottom-origin, clockwise, dark→light gradient.

## secret values

`CooldownFrame_SetDisplayAsPercentage` does `Saturate(current/max)` arithmetic in Lua, so `RenderFill` guards `issecretvalue(current)`/`issecretvalue(max)` (and `max <= 0`) before calling it. when a value is secret the swipe simply **holds its last displayed percentage** rather than advancing. XP/Honor are not currently in the secret set, but the guard keeps the bar correct if Blizzard reclassifies them.

## assets

`Core/assets/Radial/orbit-radial-{track,fill,backdrop,innervignette,border,glow,shardburst,ringcrack-light,ringcrack-heavy}.tga` plus the 6-variant `orbit-radial-crackedmetal-{1..6}.tga` durability-fracture set (512², 32-bit straight alpha, ~1 MB each). `orbit-radial-ringcrack-{light,heavy}` are the durability-gated ring-crack overlays (dark fracture lines, normal BLEND; heavy = a superset of light); `orbit-radial-shardburst` is the durability-break shrapnel (white-luminance, additive, tinted + scaled-up in the shatter flourish). generated procedurally with Pillow/numpy from `_scratch_renown/gen_v5.py` (the radial set) and a soft gaussian disc (`orbit-radial-glow`, the flourish). the fill texture is white luminance (gloss + dark→light angular gradient baked in) so `SetSwipeColor` can tint it any mode color; the glow is white so `SetVertexColor` tints it. `orbit-radial-backdrop` is the fill with its angular gradient normalised away (static white-luminance ring) so the tinted backdrop doesn't brighten around the circle — derived directly from `orbit-radial-fill.tga` (per-angle luminance flattened, gloss + alpha kept).

**filtering** — the track and border are set with `SetTexture(path, nil, nil, "TRILINEAR")` so mipmap filtering minifies the high-res art cleanly at low Scale (default `LINEAR` aliases / looks "pixelly" when a 512² texture is shown at ~70px). the fill is a Cooldown swipe and `SetSwipeTexture` has no filterMode arg, so to give it a real mip chain the fill ships as **`orbit-radial-fill.blp`** — an uncompressed RAW3 BLP2 with a full mip chain (512→1), generated by `_scratch_renown/tga2blp.py`. WoW prefers the `.blp` over the same-named `.tga` on the extensionless path, so the swipe samples mipmapped art and stays crisp from 30% to 100% Scale; the `.tga` is kept as a fallback (delete the `.blp` to revert). source resolution stays **512²** for every asset — the orb only ever displays ≤120px, a deep minification, so the *mip chain* (not raw source pixels) is what keeps it crisp, and 512² gives `shardburst` the headroom it needs when its animation scales it large.

## settings (Layout / Color / Behaviour tabs)

- **Layout**: `Scale` (%) via `SchemaBuilder:AddSizeSettings(..., scaleParams)` → `frame:SetScale`. the widget is square art with fixed `BASE_SIZE` geometry, so it sizes by scale (like MinimapButton / TalkingHead / BagBar), not a pixel width/height — `CreateScaleOnChange` re-centers the frame on scale change. the Scale range is **30–100% in steps of 5** (default 100); mipmapped (TRILINEAR) source art keeps it crisp across that range.
- **Color** (all `colorcurve`, `singleColor`): `XPColor` (XP fill), `RestedColor` (Rested XP band), `HonorColor` (Honor fill), `RepColor` (Reputation fill), `BackdropColor` (the unfilled groove). fill colours apply per-mode in `RenderFill`; the backdrop tint applies on settings change.
  - **why the backdrop has its own ring**: `orbit-radial-track` is baked near-black (mean RGB ≈ 14, max luminance 28), and `SetVertexColor` *multiplies* — so the track art can only ever be darkened, never recoloured. So `BackdropColor` instead tints `frame.BackdropRing` (drawn one sublevel above the track, below the sweep). it uses a dedicated **`orbit-radial-backdrop`** asset: the white-luminance `fill` art with its dark→light **angular sweep flattened out** (per-angle luminance normalised; radial gloss + flat alpha kept), so the backdrop is a *static* uniform ring while only the progress sweep brightens around the circle. default is a dim neutral so the groove reads dark out of the box (tinting a white ring with white would glow bright).
- **Behaviour**: `ReplaceVaultToast` (Great Vault popup), `ReplaceSocialToast` (BN/social toasts), `ShowMailToast` (new-mail toast), `ReplaceLootToast` (boss/personal loot reel), `ReplaceLootRoll` (group-loot roll panels), `ShowMilestones`, `ShowRewardToasts` — checkboxes, default on; `ShowCenterNumber` (shows the bare idle level in the hollow centre — the renown / honor / character-level **number** via `record.numeral`, falling back to `record.level`'s label for non-numeric reps like a standing); plus `Animation` (dropdown: None / Slide / Rotate Slide / Fade, default None) — the entrance below.
  - **the at-rest / hover+Shift sources (`PrimarySource` / `SecondarySource`) are NOT on this tab** — they are set only from the **right-click source menu** (`SourceMenu.lua`), the single UI for them. the settings keys + defaults (primary `auto`, secondary `honor`) and `ResolveMode` are unchanged.

## reveal animation

`Animation.lua` is a **hover/event reveal**, modelled on Orbit-Dock-Portal's `PortalReveal`: with an animation selected the orb is **concealed at rest** (slid off / rotated / alpha 0) and **reveals** when the cursor enters (`OnEnter` → `RevealOrb`) or a toast fires (`_EnterEvent`), then **reverses** on mouse-leave (`OnLeave` → `ConcealOrb`) / event end (`_ExitEvent`). it animates **`frame.Content`** — a wrapper holding every visual — never the frame itself, so the orb's EditMode / `RestorePosition`-owned anchor is untouched (Portal's content-vs-dock split). `frame` keeps mouse enabled at its resting footprint, so hovering the orb's home spot reveals it even while invisible.

a manual `OnUpdate` tween drives `progress` toward a target (1 = revealed, 0 = concealed) over `ANIM_DURATION`. `RestingTarget` = revealed while **Off**, in **Edit Mode**, **hovered**, or **showing a toast** (`_event ~= nil`); concealed otherwise. `ApplyAnimationState` (called from `ApplySettings`) snaps to that resting state on load / settings change — so picking an animation hides the orb immediately, and setting it back to **Off** shows it again.

- **Slide** — content slides between rest and an offset (also fading). **screen-edge aware**: `HiddenOffset` reads the orb's centre in physical pixels vs the screen thirds, so an orb near the **top-left** hides up-and-left (revealing toward the bottom-right), one near the **left edge** hides/reveals along the left, etc.; a dead-centre orb hides downward.
- **Rotate Slide** — the slide plus a spin: the ring textures (`Track`/`BackdropRing`/`Border`) and both `Cooldown` fills rotate by `MAX_SPIN` while concealed, unwinding to rest (the fills restore `frame._fillRotation`).
- **Fade** — content alpha only, no movement.

## notes / future

- honor as a source requires honor to exist; at very low level `UnitHonorMax` can be 0 and the sweep holds the last value (color still swaps).
- there is no tooltip hint naming the Shift (secondary) source — left out for now; the right-click menu shows the current selection (the active radio) instead.
- the rotation/reverse pair is tuned to the baked asset; if an in-game `/reload` shows the sweep mirrored or starting at the wrong origin, flip `FILL_REVERSE` or adjust `FILL_ROTATION`.
- the hollow center is deliberately contentless. a level-up animation (model/AnimationGroup) should parent to `frame.Center`.
